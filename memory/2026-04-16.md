# 2026-04-16 - 测试用例学习固化

## 用户指令
- 靓仔要求我“执行学习”
- 目标是把测试点→测试用例→交付格式这条线固化下来，并采用更主动改进的工作方式

## 本次固化的工作原则

### 1. 默认判断输入到底是“测试点”还是“正式用例”
- 如果用户给的是模块树、功能节点、场景分支，大概率先视为测试点/测试思路，不直接当正式用例
- 要先补齐：正向场景 / 异常场景 / 相关影响

### 2. 默认把测试点转成正式用例表达
- 用例名称必须是一句话概括“动作 + 场景 + 目标”
- 不能直接照抄树节点名（如“成功”“失败”“状态显示”）当用例名
- 名称尽量简洁，不写细节步骤

### 3. 默认按靓仔的四级优先级体系判断
- P0：核心主流程，失败影响多个功能
- P1：核心正向业务流，高频使用
- P2：异常场景、次要功能、扩展场景
- P3：边界/UI/低频特殊场景

### 4. 默认检查用例质量
- 步骤描述与预期结果要一一对应
- 模块分类尽量稳定，通常 4~5 级足够
- 优先级不能凭感觉乱分，要按影响度统一
- 输出时尽量避免“测试点味很重、用例味不够”的初稿

### 5. 默认提供更落地的交付形式
- 如果用户要“生成测试用例”，优先考虑同时给：
  - 可继续编辑的 Excel / xlsx
  - 适合飞书表格导入或粘贴的结构化内容
- 如果上传飞书失败，要明确说清权限/能力限制，并给替代交付方案

### 6. 默认主动改进，而不是停在执行
- 做完后主动分析：哪里只是测试点、哪里需要补异常、哪里需要补相关影响
- 给出下一步最有价值动作，而不是只说“已完成”
- 遇到同类任务，优先输出更成熟的正式版本

## 对我自己的提醒
- 不要把“能生成表格”误当成“内容已经成熟”
- 先把内容质量做好，再谈格式和上传
- 如果工具写飞书受限，及时切换为本地文件 + 可导入文本的双交付
