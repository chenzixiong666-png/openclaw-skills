# 测试日报/周报写作技能

## 适用场景

当用户提出以下需求时使用本技能：
- 写测试日报
- 写测试周报
- 补充今天的测试日报
- 汇总本周投入
- 把零散任务整理成日报/周报
- 优化日报或周报模板
- 建立长期使用的测试记录模板

## 核心规则

### 1. 测试日报模板

固定字段：
- 日期
- 测试任务
- 所属项目/模块
- 耗时
- 当前状态
- 备注

固定要求：
- 不使用“测试时间”列
- 不使用“输出物”列
- 耗时统一使用 `d`（人天）
- 最新日期内容排在最前面
- 标题格式固定为：`2026-04-24（周五）`

标准模板：

```markdown
## 2026-04-24（周五）

| 日期 | 测试任务 | 所属项目/模块 | 耗时 | 当前状态 | 备注 |
|---|---|---|---|---|---|
| 2026-04-24 | 示例任务 | 项目 / 模块 | 0.1d | 已完成 | 备注 |

## 当日汇总
- 今日总耗时：**0.1d**
- 今日主要内容：
  - xxx
- 当前问题/风险：
  - xxx
- 明日计划：
  - xxx
```

### 2. 测试周报模板

固定结构：
1. 本周工作内容
2. 本周投入汇总
3. 本周产出
4. 问题与风险
5. 下周计划

标题格式：
- 内容标题：`2026年04月20日-04月26日`
- 文档标题：`2026年04月20日-04月26日-测试周报`

投入汇总表标准模板：

```markdown
| 项目/方向 | 耗时 | 具体内容 |
|---|---:|---|
| 天气站 | 2d | 1. xxx；2. xxx |
| 9.26 | 2.1d | 1. xxx；2. xxx |
| **合计** | **4.1d** | — |
```

### 3. 测试日报文档标题规则

- 周维度文档标题使用：`2026年04月20日-04月26日-测试日报`
- 文档内部按日期分节
- 最新日期排在最前面

## 工作方法

### 用户给出零散信息时
如果用户只给：
- 若干任务
- 项目名
- 耗时
- 少量背景

则不要卡住，直接整理为：
1. 日报表格 + 当日汇总
或
2. 周报工作内容 + 投入汇总表 + 周报正文

### 日报转周报规则

当用户已经连续提供多天日报，或已提供一周内多组任务记录时：
- 先按项目/方向归并相近任务
- 再累计各项目投入，形成“本周投入汇总”
- 再提炼“本周产出”“问题与风险”“下周计划”
- 周报正文优先保留结果导向，不机械照搬每日表格

转换原则：
- 日报是明细
- 周报是汇总
- 周报中不重复堆砌过细动作，优先提炼阶段性成果、协作事项和交付风险

### 任务归类规则

整理周报时，优先按以下维度归类：
- 项目名（如天气站、9.26）
- 团队事务（如测试一组能力建设）
- 管理/协作事项（如组会、OKR、复盘）

如果用户未明确分类，则按“项目优先、公共事务兜底”的原则归类。

### 典型任务映射示例

- `V17固件第三轮测试` → `天气站`
- `V18固件主功能测试` → `天气站`
- `天气站软件1.6需求会议评审` → `天气站`
- `实现方案讨论` → `天气站 / 需求协作`
- `排期及测试交付时间讨论` → `天气站 / 需求协作`
- `9.26 hub3支持天气站matter子设备+button按键` → `9.26`
- `9.26探索测试` → `9.26`
- `客诉处理` → `测试一组能力建设`
- `Q1进展复盘填写` → `测试一组能力建设`
- `Q2个人OKR填写` → `测试一组能力建设`
- `组会` → `测试一组能力建设`

### 耗时处理
- 优先保留用户原始 `d` 口径
- 如果用户明确要求统一成 `d`，则整篇都用 `d`
- 避免 `h` 与 `d` 混用
- 若历史旧内容为 `h` 且用户要求统一，则可以在整理时直接转成近似 `d` 口径，保持整篇一致

### 排序规则
- 最新日报日期排最前
- 最新周报内容排最前
- 如果同一文档持续追加，新增内容默认插到已有内容前面，而不是后面
- 如果用户要求“调整顺序”，优先使用整体重写方式保证顺序一致，避免局部插入导致格式混乱
- 新建的日报或周报文档，在索引或汇总展示中也应排最前

## 输出风格

- 直接给结果，不写多余寒暄
- 表格字段统一，命名稳定
- 周报文案偏正式，日报偏简洁
- 若用户信息不完整，先按现有信息整理可用版本，不因缺少细节而卡住
- 提醒只在必要时出现，默认以直接产出为主

## 输入容错规则

当用户输入存在以下问题时，仍应继续整理，不要因为格式不标准而中断：
- 编号不连续（如 1、2、5）
- 中英文标点混用
- 项目名大小写或写法不统一（如 `OKR` / `okr`）
- 耗时写法不统一（如 `0.1d`、`0.1 d`、`耗时0.1d`）
- 一句话里混有任务名、结论、项目名

处理原则：
- 先提取任务本体
- 再识别项目/模块
- 再识别耗时
- 缺失信息用最小必要假设补齐
- 只要不影响理解，就不要反复追问用户

## 周报文风分级

### 1. 默认版
适用于用户日常自用，特点：
- 简洁
- 信息完整
- 结构标准

### 2. 正式汇报版
适用于发给上级、用于较正式汇报，特点：
- 更强调结果与产出
- 句式更完整
- 风险与下周计划表达更明确

### 3. 精简版
适用于快速同步，特点：
- 只保留核心投入、核心产出、风险点
- 适合 IM 消息或简短周会同步

如果用户未指定风格：
- 日报默认使用简洁版
- 周报默认使用正式版

## 推荐状态值

- 已完成
- 进行中
- 阻塞
- 待验证

- 当日汇总

### 写周报
把用户本周任务整理成：

- 本周产出
- 问题与风险

当用户只关心“本周投入”时，输出：

- 耗时
- 具体内容
- 合计

### 调整旧文档顺序
如果用户要求“最新在前”，则将已有日报/周报内容整体改写为倒序排列，保证最新日期或最新内容显示在最上方。

### 模板优化
如果用户持续调整字段：
- 优先删减字段，保持模板轻量
- 将最新确认的规则覆盖旧规则
- 保持日报和周报的命名、字段、排序一致
