## 测试用例编写原则 ⭐ 核心规则

**用户的测试思想体系：**

1. **覆盖链条完整**
   - 需求 → 测试点 → 测试用例
   - 每个测试用例必须对应具体测试点
   - 每个测试点必须对应需求条目
   - 不能有孤立的测试用例或测试点

2. **测试思路三维度**
   - **正向场景**：功能正常工作的路径
   - **异常场景**：错误输入、边界、异常状态的处理
   - **相关影响**：对其他模块/功能的影响，副作用检查

3. **质量要求**
   - 严谨、完整、有逻辑
   - 不是随意写，要有体系

## 测试用例等级评定标准 ⭐ 四级体系

### **P0 级 - 冒烟级（烧烤测试）**

| 维度 | 内容 |
|------|------|
| **划分原则** | 核心功能的主流程和基本业务流，保证功能完整性 |
| **划分依据** | 失败会导致**多个功能无法运行**（如：扫地机-配网、建图、清扫/暂停/回充、传感器等） |
| **设计方法** | 场景法 |
| **重要性** | 🔴 **最高** - 失败导致其他多处功能无法使用，无法进行后续测试 |
| **执行频率** | ⚡ **每一轮版本系统测试都必须执行** |

### **P1 级 - 核心功能级**

| 维度 | 内容 |
|------|------|
| **划分原则** | 核心功能的正向业务流，覆盖平台所有重要业务流 |
| **划分依据** | 功能交互相关、应用场景多、使用频率高的正常功能（如：地图编辑、禁区/虚拟墙编辑、档位选择等） |
| **重要性** | 🟠 **很高** - 覆盖所有功能的正向业务流，保证软件功能稳定使用 |
| **执行频率** | 📋 **非回归的系统测试版本基本都需要验证** |

### **P2 级 - 功能扩展级**

| 维度 | 内容 |
|------|------|
| **划分原则** | 重要功能的正向业务流 + 核心/重要功能的异常业务流 |
| **划分依据** | 使用频率低于P1的场景（如：集尘设置、清洁设置等），以及异常场景（机器各异常状态、重定位失败交互等） |
| **重要性** | 🟡 **中等** - 非产品基本功能，不常用应用场景、非典型参数配置，体现容错性、适应性、组合性 |
| **执行频率** | 🔍 **适合性、可靠性、性能、压力、安全性相关用例** |

### **P3 级 - 边界/UI级**

| 维度 | 内容 |
|------|------|
| **划分原则** | 其他异常用例、UI/UE 相关用例、边界特殊场景 |
| **划分依据** | 产品功能边界和特殊场景逆向用例，测试条件苛刻、难以构建的场景 |
| **重要性** | 🟢 **低** - 客户使用频率非常低 |
| **执行频率** | 📍 **有选择性执行，通常按需** |

| 是UI/UE相关吗？ | ❌ | ❌ | 一般 | ✅ |

## 用例模块和用例名称规范 ⭐ 命名标准

### **用例名称要求**

1. **高度概括** 
   - 用一句话总结这个用例在做什么
   - 不要写细节步骤，只要表达核心意图
   - 例子：❌ "打开App → 进入设置 → 点击清洁模式 → 选择强力模式"
   - 例子：✅ "设置清洁模式为强力模式"

2. **用例名唯一**
   - 同一模块内不能有重复的用例名
   - 名称必须能够区分不同的测试点

3. **多场景用例用（）表示**
   - 如果一个用例涉及多个场景或多种情况，用括号来体现
   - 格式：`用例核心名称（场景1/场景2/场景3）`
   - 例子：✅ "验证扫地机清扫模式（标准模式/强力模式/静音模式）"
   - 例子：✅ "测试禁区设置（矩形禁区/圆形禁区）"

### **设计原则**

- **简洁清晰** - 看名称就知道用例的目的
- **不包含步骤** - 步骤在用例详情里，名称里只要目标
- **便于分类** - 名称要能支撑用例的组织和查找

## 用例编写实战规范 ⭐ 学自Climate Panel用例

### 表格结构（列）

| 列名 | 说明 | 示例 |
|------|------|------|
| ID | 用例编号 | T193014 |
| 一级模块~七级模块 | 用例分类树（最多7级） | 固件用例 > 添加、删除 > ... |
| 用例名称 | 简洁的一句话表达 | "删除设备后再添加" |
| 优先级 | P0/P1/P2/P3 | P0, P1 |
| 用例类型 | 功能测试、性能测试等 | 功能测试 |
| 前置条件 | 执行用例前的环境状态 | "账号1已添加设备，已将家庭分享给账号2" |
| 步骤描述 | 分步骤的操作步骤，用`#`分割 | `#步骤1\n#步骤2\n#步骤3` |
| 预期结果 | 分步骤的预期结果，用`#`分割 | `#结果1\n#结果2\n#结果3` |
| 维护人 | 谁负责这个用例 | 陈梓雄 |

### 核心编写规则

1. **用例名称**
   - ✅ 简洁、高度概括（一句话说清楚）
   - ✅ 带场景时用括号：`按键背光（背光区域检查）`
   - ❌ 不包含步骤细节
   - 例子：✅ "删除设备后再添加" / ❌ "点击删除→确认→重新添加"

2. **步骤描述和预期结果**
   - 用 `#` 分割多步骤（每个 `#` 代表一个步骤）
   - 步骤数和预期结果数必须对应
   - 可以在一个步骤中描述多个操作（逗号或"和"连接）
   - 例子：
     ```
     步骤：#按ON/OFF按键\n#按开关、上下\n#按模式、风速
     结果：#背光1亮起，10s后熄灭\n#背光2亮起，10s后熄灭\n#背光3亮起，10s后熄灭
     ```

3. **前置条件**
   - 简洁描述环境状态（用逗号或文字表达）
   - 例子：
     - ✅ "设备已绑定空调，且设备为欧规设备"
     - ✅ "设备已开启人感"
     - ✅ 空白（没有前置条件）

4. **模块分类（一级~七级）**
   - 最多用到4-5级就够了
   - 模块树要合理，同一个模块下的用例数量不要太悬殊
   - 按功能域或业务流组织

5. **优先级判断**
   - P0：核心功能主流程，失败影响多个功能
   - P1：功能正向业务流，使用频率高
   - P2：异常场景或次要功能
   - P3：边界、UI、特殊场景

| **艺术画框** | next / previous / uploadImage | 支持 URL 和 Base64 |

### ⚙️ 三种控制方式

1. **直接命令**（大多数设备）
   - 立即执行，HTTP 响应返回结果
   - 例：turnOn, setPosition, lock

2. **异步命令**（某些蓝牙设备和 Keypad）
   - 执行后返回 taskId，需配置 webhook 获取结果
   - 例：Keypad 创建/删除密码

3. **Scenes 兜底**
   - 当设备无直接命令支持时，用 Scenes 代替
   - CLI 会自动检测能力并给出提示

### 🏠 设备/家庭/房间层级

运行 `node switchbot_cli.js list` 会返回：
```
🏠 Home（家庭名称）
  └─ 未分配房间: 设备A, 设备B
  └─ 客厅: 设备C, ...
```

红外设备（如空调）只有 hubDeviceId，需要关联到对应的 Hub 来确定所属家庭。

### ✅ 设备预检机制

CLI 脚本会自动检查：
- 对于 BLE 设备（Bot/Lock/Curtain）：需要 Hub + Cloud Services 启用
- 若检查失败会给出明确提示，要求在 SwitchBot App 中绑定 Hub

### 🔒 安全与敏感操作

- 不要在日志中输出 token/secret
- unlock 等敏感操作前建议二次确认
- 环境变量通过服务器安全注入（不硬编码）

### 🧠 实战要点

**参数格式容易出错的地方：**
1. **窗帘位置**：`setPosition --pos=50` 自动转成 `0,ff,50` 格式
2. **RGB 颜色**：`"255:100:0"`（R:G:B）
3. **空调命令**：`"温度,模式,风速,电源"` — 模式 2=冷却, 5=加热
4. **扫地机**：使用 JSON 参数 `{"action":"sweep_mop","param":{"fanLevel":2,"waterLevel":1}}`
5. **Keypad**：密码必须 6-12 位数字，时间戳用 10 位 Unix 时间
6. **温控器**：模式范围 0-5，温度范围 4-35°C

**排查思路：**
- 160 错误 = 设备不支持该命令，改用 Scenes
- 190/100 错误 = token/secret 问题或时间同步偏差
- 预检失败 = Hub 未绑定或 Cloud Services 未启用
